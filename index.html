Sencha - the bad parts

By Dominykas Blyžė (@dymonaz), 2012-05-01 for DublinJS

Disclaimer

These are just feelings from a few months of tinkering.

I'm probably doing it wrong.

GPL

Compare to Meteor who changed GPL to MIT

No progressive enhancement

  • Unsuitable for general web
  • If it's only suitable for apps - may as well go native!

Poor history support

The Web is URL not HTML. A native app which uses URLs as input/output is more "Web" than an HTML app for which the only URL is its home page

William Vambenepe, @vambenepe

Poor history support

On Navigation View:

It's not a bug, that back button doesn't do anything with history by design.

Mitchell Simoens, Sencha Inc, Senior Forum Manager

TDD is at the very least painful

  • Framework itself feels too tightly coupled
  • Too many dependencies - need to learn the framework inside out
  • Long hierarchy chains
  • Writing more config than code - hard to test

TDD not advocated

  • They recommend jasmine - but examples are integration testing
  • They recommend phantomjs - but things need to run in the environments they need to run
  • Source code itself delivered without tests
  • Examples used to have tests folders... which were empty

Lots of UI components do not a great app make

A kitchensink can lead to designs which reflect system's internals and are not relevant to users' mental models.

Hard to debug: HTML

Hard to debug: JS

Debugging JS with Sencha

  • There's a Firebug plugin
  • ...costs 25USD
  • ...for the first year
  • ...Sencha Touch only supports webkit
  • WTF does one do on a mobile phone?

Debugging: learning curve

Looking at the source doesn't help due to dependencies, coupling and, well, spaghetti.

580kb (minified) is a lot to look through...

580kb is also a lot to load

Similar functionality apps running on Android - 40Mb with Sencha, 15Mb with jQuery mobile

BlackBerry emulator just crashes...

Error handling is hard

Data stores are awesome, but the base class doesn't even have events for errors

On mobile one MUST handle errors, build retry capabilities, build offline capabilities, build smart preload capabilities

Gut feeling...

If there are multiple ways to do things, Sencha devs will strive to choose the opposite of the Right Way.

Example: file:// + AJAX + HTTP status codes

What it's good for?

  • Limited use internal apps in controlled environments
  • Prototypes
  • NOT for prime-time apps, unless you have the time to learn it properly

But why learn a specific framework, when you can rely on The Web?

</rant>
